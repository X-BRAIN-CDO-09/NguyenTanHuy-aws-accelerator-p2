1. Dự án này dùng để làm gì?

Dự án dùng Terraform để tự động dựng một hệ thống Kubernetes nhỏ trên AWS. Người dùng chỉ chạy một lệnh bash scripts/up.sh. Terraform tạo hạ tầng AWS, EC2 tự cài Docker, Minikube, Kubernetes, deploy app HTML/CSS vào Pod nginx, sau đó public app ra Internet bằng AWS Application Load Balancer thật.

Điểm quan trọng: app không cài thẳng trên EC2. EC2 chỉ là host để chạy Docker và Minikube. App thật sự chạy trong Kubernetes Pod.

3. Hai Terraform provider dùng để làm gì?

AWS Provider

Quản lý hạ tầng thật trên AWS: EC2, Security Group, ALB, Target Group, Listener, Key Pair.

TLS Provider

Sinh SSH private/public key trong Terraform. Public key được AWS dùng để tạo EC2 Key Pair.

Hai provider liên kết với nhau như thế nào?

resource "tls_private_key" "minikube" {
  algorithm = "RSA"
  rsa_bits  = 4096
}

resource "aws_key_pair" "minikube" {
  key_name   = "${var.project_name}-key"
  public_key = tls_private_key.minikube.public_key_openssh
}

Đây là wire provider: output của TLS provider là public_key_openssh, sau đó AWS provider lấy giá trị đó làm input để tạo aws_key_pair.

4. Khi chạy bash scripts/up.sh thì cái gì chạy trước?

Script bắt đầu

up.sh xác định ROOT_DIR, TF_DIR và tạo thư mục .generated.

terraform init

Tải provider hashicorp/aws và hashicorp/tls.

terraform apply

Tạo TLS key, AWS KeyPair, EC2, Security Group, ALB, Target Group, Listener.

TLS chạy

tls_private_key.minikube sinh private/public key.

AWS tạo EC2

EC2 được gắn user_data.sh; khi boot lần đầu, cloud-init tự chạy file này.

user_data.sh chạy trên EC2

Cài Docker, kubectl, Minikube, start Minikube, deploy app vào Kubernetes.

up.sh lấy private key

Chạy terraform output -raw private_key_pem và ghi vào .generated/minikube-key.

up.sh chờ EC2 sẵn sàng

SSH vào EC2, chạy cloud-init status --wait và kiểm tra /var/log/k8s-app-ready.log.

In ALB URL

Khi xong, script in ALB URL để mở web.
bash scripts/up.sh
  ├── terraform init
  ├── terraform apply -auto-approve
  │     ├── tls_private_key.minikube
  │     ├── aws_key_pair.minikube
  │     ├── aws_security_group.*
  │     ├── aws_instance.minikube
  │     │     └── cloud-init chạy user_data.sh
  │     ├── aws_lb.app
  │     ├── aws_lb_target_group.app
  │     └── aws_lb_listener.http
  ├── terraform output -raw private_key_pem > .generated/minikube-key
  ├── ssh vào EC2 để chờ cloud-init + k8s-app-ready.log
  └── in ALB URL

5. File nào cài Docker, Minikube và Kubernetes?

File chịu trách nhiệm cài software trên EC2 là terraform/user_data.sh. File này được AWS EC2 cloud-init chạy tự động khi instance boot lần đầu.

5.1 Cài Docker trên EC2

apt-get update
apt-get install -y ca-certificates conntrack curl docker.io gnupg socat
systemctl enable --now docker
usermod -aG docker ubuntu

Đoạn này cài Docker Engine, bật Docker service và thêm user ubuntu vào group docker.

5.2 Cài kubectl

apt-get install -y kubectl

kubectl là CLI để gửi lệnh vào Kubernetes API Server.

5.3 Cài Minikube

curl -Lo /usr/local/bin/minikube https://storage.googleapis.com/minikube/releases/latest/minikube-linux-amd64
chmod +x /usr/local/bin/minikube

5.4 Start Minikube bằng Docker driver

minikube start   --driver=docker   --container-runtime=docker   --cpus=2   --memory=1800mb   --ports=30080:30080

Có, Minikube của dự án chạy trên Docker. Đường đi là:

EC2 Ubuntu
  → Docker Engine
  → Minikube node/container
  → Kubernetes cluster
  → Pod nginx

5.5 Tạo Kubernetes manifest

user_data.sh tự tạo file /home/ubuntu/html-app.yaml, trong đó có Namespace, ConfigMap, Deployment và Service.

5.6 Deploy app vào Kubernetes

kubectl apply -f "$HOME/html-app.yaml"
kubectl rollout status deployment/html-app -n app --timeout=180s
kubectl get pods -n app
kubectl get svc -n app

5.7 Báo hiệu app sẵn sàng

cp /var/log/k8s-app-deploy.log /var/log/k8s-app-ready.log

File /var/log/k8s-app-ready.log là dấu hiệu để up.sh biết bootstrap đã xong.

6. Đường đi request khi mở website

Browser gọi ALB URL

Người dùng mở URL dạng http://k8s-aws-1click-alb-xxxx.us-east-1.elb.amazonaws.com.

ALB nhận HTTP port 80

Security Group của ALB cho phép inbound HTTP từ Internet.

Listener forward

Listener port 80 của ALB forward request vào Target Group.

Target Group gọi EC2:30080

Target type là instance, port backend là 30080.

NodePort nhận traffic

Port 30080 là NodePort của Kubernetes Service.

Service gọi Pod

Service chọn Pod có label app: html-app.

Pod trả HTML/CSS

Nginx trong Pod serve nội dung từ ConfigMap mount vào /usr/share/nginx/html.

7. Vì sao app chạy trong Kubernetes chứ không phải trên EC2?

Trên EC2 không cài nginx trực tiếp bằng apt install nginx. Nginx chỉ chạy trong container thuộc Kubernetes Pod:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: html-app
  namespace: app
spec:
  replicas: 2
  template:
    spec:
      containers:
        - name: nginx
          image: nginx:alpine
Kiểu không đạtKiểu dự án đang làm
EC2 chạy nginx trực tiếpKubernetes Pod chạy nginx
ALB trỏ vào app host EC2ALB trỏ vào Service NodePort 30080
Không có self-healingDeployment tự tạo lại Pod nếu Pod chết

Cách chứng minh

ssh -i .generated/minikube-key ubuntu@<EC2_PUBLIC_IP>
docker ps
minikube status
kubectl get pods -n app
kubectl get svc -n app
